\section{خلاصه وضعیت فعلی TAFlow}
پروژه \code{TAFlow} یک سامانه full-stack برای مدیریت دستیاران آموزشی است. بر اساس ساختار فعلی ریپازیتوری، پروژه شامل Next.js App Router، TypeScript، Prisma/PostgreSQL، Auth.js/NextAuth، Zod، Argon2، TOTP 2FA، Docker Compose، صفحات RTL فارسی، APIهای متعدد، ماژول‌های پیام‌رسانی، درخواست TA، گواهی، نظرسنجی، دفتر نمره و گزارش‌گیری است. این سطح از پیاده‌سازی برای MVP بسیار خوب است؛ اما برای نسخه پایدار و قابل دفاع باید چند دسته اصلاح جدی انجام شود:

\begin{enumerate}
  \item تثبیت محیط توسعه، package manager و نسخه‌ها.
  \item اصلاح دیتابیس و constraintها.
  \item تکمیل workflowهای Head TA، دفتر نمره، نظرسنجی، گواهی و جلسات رفع اشکال.
  \item افزایش امنیت، audit، rate limiting و تست permissionها.
  \item بازطراحی UI/UX به شکل مدرن، گرافیکی، role-based، responsive و قابل استفاده.
  \item اضافه‌کردن انیمیشن‌های کنترل‌شده، اسکرول مدرن، micro-interaction و جلوه‌های landing page بدون تخریب performance و accessibility.
\end{enumerate}

\section{چک‌لیست کامل ۷۵ اصلاح و بهبود}

\subsection{الف) اصلاحات حیاتی زیرساخت و نسخه‌ها}
\begin{enumerate}
  \item \textbf{یکسان‌سازی package manager:} پروژه نباید هم‌زمان \code{package-lock.json} و اجرای \code{pnpm} داشته باشد. تصمیم نهایی: استفاده از \code{pnpm}. حذف \code{package-lock.json}، ساخت و commit کردن \code{pnpm-lock.yaml}، و اضافه‌کردن \code{"packageManager": "pnpm@11.9.0"} به \code{package.json}.
  \item \textbf{قفل کردن نسخه dependencyها:} نسخه‌های اصلی مثل \code{next}، \code{react}، \code{react-dom}، \code{prisma}، \code{@prisma/client}، \code{next-auth}، \code{typescript} و \code{zod} باید بدون \code{^} یا \code{~} ثبت شوند تا نصب‌های بعدی پروژه را خراب نکند.
  \item \textbf{تثبیت Prisma روی نسخه سازگار:} فعلاً پروژه باید روی \code{prisma@6.19.3} و \code{@prisma/client@6.19.3} pin شود؛ چون schema فعلی با Prisma 7 سازگار نیست. در آینده migration به \code{prisma.config.ts} انجام شود.
  \item \textbf{حذف وابستگی به config deprecated:} بخش \code{package.json#prisma} برای seed در بلندمدت به \code{prisma.config.ts} منتقل شود تا در Prisma 7 خطا ایجاد نکند.
  \item \textbf{نوشتن راهنمای troubleshooting:} README باید خطاهای رایج مثل \code{Prisma 7 datasource url error}، \code{pnpm approve-builds}، کندی npm، conflict پورت PostgreSQL، reset دیتابیس و seed دوباره را توضیح دهد.
\end{enumerate}

\subsection{ب) اصلاحات دیتابیس و Prisma schema}
\begin{enumerate}[resume]
  \item \textbf{اصلاح unique در CourseRoleAssignment:} constraint فعلی \code{@@unique([courseOfferingId, userId, role])} با تاریخچه revoke سازگار نیست. باید حذف شود و در migration دستی یک partial unique index فقط برای نقش‌های فعال ساخته شود:
\begin{latin}\begin{lstlisting}
CREATE UNIQUE INDEX course_role_active_unique
ON "CourseRoleAssignment" ("courseOfferingId", "userId", "role")
WHERE "revokedAt" IS NULL;
\end{lstlisting}\end{latin}
  \item \textbf{اصلاح رأی‌گیری ناشناس PollVote:} unique روی \code{[pollId, voterId]} برای \code{NULL} کافی نیست. باید \code{respondentHash} برای anonymous vote اجباری شود و partial unique index اضافه شود:
\begin{latin}\begin{lstlisting}
CREATE UNIQUE INDEX poll_vote_hash_unique
ON "PollVote" ("pollId", "respondentHash")
WHERE "respondentHash" IS NOT NULL;
\end{lstlisting}\end{latin}
  \item \textbf{اصلاح مدل SurveyAnswer:} برای جلوگیری از پاسخ تکراری، مدل باید دو مرحله‌ای شود: \code{SurveyResponse} و \code{SurveyAnswer}. هر response یکبار برای یک survey و هر answer یکبار برای یک question باشد.
  \item \textbf{جلوگیری از evaluation تکراری:} برای \code{ProfessorEvaluation} و \code{TAEvaluation} باید unique بر اساس \code{courseOfferingId}، ارزیاب و فرد ارزیابی‌شونده اضافه شود. در حالت anonymous از \code{respondentHash} استفاده شود.
  \item \textbf{CHECK constraint برای ratingها:} امتیازهای professor/TA/session/survey باید در دیتابیس بین ۱ تا ۵ یا بازه تعریف‌شده باشند.
  \item \textbf{CHECK constraint برای نمره:} در \code{GradeRecord} مقدار \code{score} نباید منفی یا بیشتر از \code{maxScore} باشد، مگر اینکه bonus به صورت صریح طراحی شود.
  \item \textbf{کنترل وزن GradeCategory:} مجموع وزن categoryهای یک درس باید در service به ۱۰۰ درصد محدود شود یا طبق سیاست پروژه کنترل شود.
  \item \textbf{کنترل بازه تاریخ‌ها:} برای \code{OfficeHourSession}، \code{TAOpportunity}، \code{AcademicCalendarEvent}، \code{AvailabilityPoll} و بازه‌های گواهی باید \code{start < end} enforce شود.
  \item \textbf{اصلاح timezone پیش‌فرض:} اگر مخاطب فارسی/ایرانی است، پیش‌فرض از \code{Asia/Baku} به \code{Asia/Tehran} یا بهتر از آن به \code{APP_DEFAULT_TIMEZONE} قابل تنظیم در env تغییر کند.
  \item \textbf{تاریخ شمسی و UTC:} ذخیره دیتابیس باید UTC بماند، اما UI باید Jalali date picker، نمایش شمسی، timezone-aware formatting و export دوگانه شمسی/میلادی داشته باشد.
  \item \textbf{سیاست soft delete:} برای user، file، message، grade، certificate و application مشخص شود که delete واقعی مجاز است یا فقط archived/revoked/deletedAt.
  \item \textbf{تکمیل idempotent seed:} همه داده‌های seed با \code{upsert} ساخته شوند یا با \code{seedTag} قابل پاک‌سازی باشند تا اجرای چندباره seed باعث duplicate نشود.
\end{enumerate}

\subsection{ج) Auth، RBAC و امنیت}
\begin{enumerate}[resume]
  \item \textbf{تست کامل permissionها:} برای همه routeهای حساس سناریوهای 401، 403، role اشتباه، role منقضی، role revoke شده، student دسترسی به professor route، و TA ویرایش grade غیرمجاز تست شود.
  \item \textbf{Rate limit production-grade:} rate limit فقط در memory نباشد؛ با Redis/Upstash/Redis Docker پیاده شود. مسیرهای login، register، forgot-password، message، file upload، application، survey، poll و certificate محدود شوند.
  \item \textbf{تکمیل سیاست SSO/Keycloak:} برای SSO باید allowed domains، وضعیت email verification، نقش پیش‌فرض، رفتار user suspended/deleted و mapping نقش‌ها مشخص شود.
  \item \textbf{Audit log اجباری:} role assignment/revoke، application accept/reject، grade change/publish، certificate issue/revoke، file download، survey results view، user suspension، 2FA disable و password reset حتماً audit شوند.
  \item \textbf{یکسان‌سازی error response:} همه APIها ساختار خطای ثابت داشته باشند: \code{ok: false}، \code{error.code}، \code{error.message}، \code{details}.
  \item \textbf{اعتبارسنجی env با Zod:} فایل \code{src/env.ts} ساخته شود و envهای حیاتی مثل \code{DATABASE_URL}، \code{AUTH_SECRET}، \code{AUTH_ENCRYPTION_KEY}، \code{NEXTAUTH_URL}، storage و SMTP validate شوند.
  \item \textbf{Security headers:} در \code{next.config.mjs} یا middleware، headerهای \code{Content-Security-Policy}، \code{X-Frame-Options}، \code{X-Content-Type-Options}، \code{Referrer-Policy}، \code{Permissions-Policy} و \code{Strict-Transport-Security} اضافه شود.
  \item \textbf{CSRF/Origin check:} برای همه mutation routeها یعنی \code{POST}، \code{PATCH} و \code{DELETE}، session و origin بررسی شود.
  \item \textbf{MFA برای نقش‌های حساس:} برای \code{PROFESSOR}، \code{EDUCATION_ADMIN} و ترجیحاً \code{HEAD_TA}، 2FA اجباری شود.
  \item \textbf{تست escalation:} سناریوهایی مثل TA که خودش را Head TA کند، student که status application خودش را approve کند، student که grade خودش را تغییر دهد، Head TA که نقش professor بدهد، و کاربر suspended با session فعال تست شوند.
\end{enumerate}

\subsection{د) گردش کار TA / Head TA}
\begin{enumerate}[resume]
  \item \textbf{هماهنگی Head TA با نیاز پروژه:} مشخص شود Head TA فقط recommendation می‌دهد یا واقعاً approve/reject می‌کند. اگر باید اختیار واقعی داشته باشد، permissionهای \code{APPROVE_TA_APPLICATION}، \code{REJECT_TA_APPLICATION} و \code{ASSIGN_TA_ROLE} اضافه شود.
  \item \textbf{درخواست جدا برای TA و Head TA:} اگر کاربر می‌تواند هم برای TA و هم Head TA یک فرصت درخواست بدهد، unique باید به \code{[opportunityId, applicantId, requestedRole]} تغییر کند.
  \item \textbf{Scoring توضیح‌پذیر:} سیستم پیشنهاد TA باید دلیل ارائه دهد: معدل، نمره درس، گذراندن درس، سابقه TA، availability، مهارت‌ها و ارزیابی‌های قبلی.
  \item \textbf{Workflow مصاحبه کامل:} وضعیت‌های shortlist، interview scheduled، interview completed، interview score، professor note، headTA note، final decision، offer sent، offer accepted/declined اضافه شود.
  \item \textbf{Appeal/review:} دانشجو بتواند درخواست بررسی مجدد یا اعتراض ثبت کند و استاد/آموزش پاسخ دهد. همه رویدادها audit شوند.
  \item \textbf{Talent Pool واقعی:} مهارت‌ها، درس‌های گذرانده‌شده، نمره‌ها، سابقه TA، ارزیابی قبلی، availability و professor favorite list اضافه شود.
  \item \textbf{Recommendation Engine:} برای هر درس، سیستم بهترین TAها را با امتیاز و دلیل پیشنهاد دهد.
\end{enumerate}

\subsection{ه) دفتر نمره، تکالیف و اعتراض نمره}
\begin{enumerate}[resume]
  \item \textbf{Assignment دقیق grade item به TA:} مدل \code{GradeItemAssignee} اضافه شود تا TA فقط grade itemهای assigned به خودش را ویرایش کند.
  \item \textbf{Regrade request:} دانشجو بتواند برای یک نمره اعتراض ثبت کند: reason، status، assignedTo، response، resolvedAt.
  \item \textbf{Rubric:} مدل‌های \code{Rubric}، \code{RubricCriterion} و \code{RubricScore} برای نمره‌دهی منصفانه‌تر اضافه شود.
  \item \textbf{Import از Excel:} علاوه بر export، import نمره با validation، preview خطاها و تایید نهایی اضافه شود.
  \item \textbf{Lock بعد از publish:} بعد از انتشار نمره، تغییر مستقیم توسط TA ممنوع شود و فقط با request و approval انجام شود.
  \item \textbf{Grade Change Workflow:} مدل \code{GradeChangeLog} باید با workflow approval کامل شود، نه فقط ثبت تغییر ساده.
\end{enumerate}

\subsection{و) جلسات رفع اشکال، Meet و حضورغیاب}
\begin{enumerate}[resume]
  \item \textbf{ثبت‌نام دانشجو برای جلسه:} مدل \code{OfficeHourRegistration} با unique روی \code{[sessionId, studentId]} اضافه شود.
  \item \textbf{Attendance:} مشخص شود چه کسانی ثبت‌نام کردند، چه کسانی حضور داشتند، چند نفر حاضر شدند و رضایت جلسه چقدر بود.
  \item \textbf{Recurring sessions:} جلسات هفتگی، تکرار تا تاریخ مشخص و cancel کردن یک occurrence اضافه شود.
  \item \textbf{Google Calendar/Meet integration:} فعلاً URL دستی کافی است، اما نسخه بهتر باید Google Calendar API، ساخت Meet link، reminder ایمیلی و iCal را کامل‌تر کند.
\end{enumerate}

\subsection{ز) گواهی TA / Head TA}
\begin{enumerate}[resume]
  \item \textbf{QR و verification page قوی:} هر گواهی باید QR code، verifyCode، صفحه عمومی verification، role، course، semester، hours/workload و issuer داشته باشد.
  \item \textbf{Certificate revocation:} امکان revoke با reason، revokedBy و revokedAt اضافه شود.
  \item \textbf{صدور گروهی گواهی:} استاد یا آموزش بتواند در پایان ترم برای چند TA/Head TA یکجا گواهی صادر کند.
  \item \textbf{قالب PDF رسمی:} قالب گواهی با نام دانشگاه، شماره رهگیری، امضای دیجیتال/تصویری مجاز، QR و اطلاعات محدود قابل چاپ طراحی شود.
\end{enumerate}

\subsection{ح) پیام‌رسانی، فایل و اعلان‌ها}
\begin{enumerate}[resume]
  \item \textbf{Unread count و seen status:} threadها باید تعداد پیام خوانده‌نشده و last read timestamp داشته باشند.
  \item \textbf{Attachment پیام:} مدل \code{MessageAttachment} برای ارسال فایل داخل گفت‌وگو اضافه شود.
  \item \textbf{ضد spam و moderation:} rate limit پیام، report message، mute/block thread و admin moderation اضافه شود.
  \item \textbf{Sanitize محتوای پیام:} اگر پیام rich text باشد، HTML حتماً sanitize شود تا XSS رخ ندهد.
  \item \textbf{Virus scan فایل:} مدل \code{UploadedFile} باید \code{scanStatus}، \code{scanAt} و \code{scanProvider} داشته باشد.
  \item \textbf{Audit دانلود فایل:} دانلود رزومه، گواهی، فایل نمره و فایل خصوصی درس باید audit شود.
  \item \textbf{Signed URL:} فایل‌ها نباید public مستقیم باشند؛ URL موقت با expiration و permission check ایجاد شود.
  \item \textbf{Notification channels:} اعلان‌ها باید channel داشته باشند: in-app، email، digest، urgent، read/unread و scheduled reminder.
  \item \textbf{Activity Timeline:} برای هر course offering timeline فعالیت‌ها نمایش داده شود: درخواست TA، بررسی، جلسه، نمره، گواهی، پیام و اطلاعیه.
\end{enumerate}

\subsection{ط) نظرسنجی، ارزیابی و حریم خصوصی}
\begin{enumerate}[resume]
  \item \textbf{آستانه حداقل پاسخ:} نتایج anonymous اگر کمتر از مثلاً ۵ پاسخ دارند، نمایش داده نشوند.
  \item \textbf{تحلیل کیفی نظرات:} دسته‌بندی نظرات به positive، negative، neutral، keyword summary و شکایت‌های پرتکرار اضافه شود.
  \item \textbf{ارزیابی دوره‌ای:} midterm evaluation، endterm evaluation، office hour feedback، TA performance review و professor evaluation اضافه شود.
  \item \textbf{Privacy-aware exports:} خروجی‌های survey/evaluation نباید respondent را افشا کنند، مگر برای admin مجاز و با audit.
\end{enumerate}

\subsection{ی) UI/UX، داشبورد و تجربه کاربری}
\begin{enumerate}[resume]
  \item \textbf{Loading/Empty/Error state:} همه صفحات باید loading skeleton، empty state، error state، retry button و permission denied state داشته باشند.
  \item \textbf{داشبوردهای role-based واقعی:} داشبورد student، TA، Head TA، Professor و Admin باید کاملاً متفاوت و متناسب با نیاز همان نقش باشد.
  \item \textbf{Command Palette permission-aware:} جستجوی Ctrl+K نباید نتیجه‌ای نشان دهد که کاربر permission دیدنش را ندارد.
  \item \textbf{Responsive mobile:} جدول‌ها، کارت‌ها، فرم‌ها، فیلترها و actionها باید روی موبایل قابل استفاده باشند.
  \item \textbf{Jalali UI:} date picker شمسی، اعداد فارسی optional، RTL calendar و خروجی تاریخ شمسی اضافه شود.
  \item \textbf{Dark mode کامل:} dark mode فقط تغییر رنگ ساده نباشد؛ همه کامپوننت‌ها، chartها، formها و statusها باید سازگار شوند.
  \item \textbf{Design system واحد:} tokenهای رنگ، radius، shadow، typography، spacing، motion و component variants باید در فایل‌های مرکزی تعریف شوند.
\end{enumerate}

\subsection{ک) DevOps، تست و مستندات}
\begin{enumerate}[resume]
  \item \textbf{Dockerfile production:} علاوه بر docker-compose dev، فایل \code{Dockerfile} و \code{docker-compose.prod.yml} اضافه شود.
  \item \textbf{Health check کامل:} health endpoint باید database، storage، mail service optional، app version و migration status را بررسی کند.
  \item \textbf{CI/CD با GitHub Actions:} pipeline شامل install، generate، typecheck، lint، test، build، Playwright و migration check اضافه شود.
  \item \textbf{Backup strategy:} برای PostgreSQL و object storage بکاپ روزانه، restore script، encryption و retention policy تعریف شود.
